Ops Demo ohne KI (12) - C# Edition
==================================

Ziel: MCP als Standard-Connector zeigen, ohne ein Modell einzusetzen. Fokus: typische C#-Use-Cases (aspnet, NuGet, Tests) mit einem ironischen Augenzwinkern. Der Host (CLI) ruft Tools/Resources bewusst selbst auf.

Vorbereitung:
- Server starten: `dotnet run` in `12/server`
- Client starten: `dotnet run` in `12/client`
- Kein LLM oder API-Key noetig. Default-Endpunkt: http://localhost:5600/sse (anpassbar via MCP_SERVER_URL).

Services (Startzustand, alle ohne KI):
- web (ASP.NET Frontend) – leicht degraded, braucht Logging-Fix.
- worker (Hangfire) – meist stabil.
- nuget (Private Feed) – fuer lokale Restores.
- build (Build Agent) – maintenance, weil neue SDKs installiert werden.
- tests (Integration Runner) – wackelt, meldet Fehler.

Ablaufvorschlag (alles manuell, gerne humorvoll kommentieren):
1. Option 1 "Status-Uebersicht lesen" -> Resource `ops/overview`.
2. Option 2 "Services listen" -> Tool `ops.services.list` (Tabelle).
3. Option 3 "Service-Details lesen" -> z. B. ID `web`, `build` oder `tests`.
4. Option 4 "Service neu starten" -> ID `tests` oder `web`; danach Option 3 erneut zum Vergleich.
5. Option 5 "Deployment markieren" -> ID `web`, Version `9.0.0-preview1-hotfix`, Notiz: "Fix nullable storm".
6. Option 6 "Timeline-Notiz" -> z. B. "dotnet restore nur mit --disable-parallel".
7. Option 7 "Timeline anzeigen" -> Resource `ops/timeline` lesen (zeigt manuelle Aktionen).
8. Option 8 "Runbook anzeigen" -> C#-Themen wie `fix-nullable-storm`, `nuget-cache-panic`, `tests-red-green` oder Klassiker `restart-service`.

Hinweis:
- Fokus: MCP-Bausteine (Tools, Resources, Runbooks) in einem reinen Operator-Flow; keine KI-Aufrufe.
- Falls Port oder Host abweichen, MCP_SERVER_URL im Client setzen.
