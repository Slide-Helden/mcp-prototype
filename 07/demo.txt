Demo 07: Testplan-Katalog (nur lesen)
=====================================
Spezialisiert | Nur Resources

WAS WIR ZEIGEN: MCP-Server kann auch rein als Daten-Provider dienen.
Hier: Testplaene als Resources ohne ausfuehrbare Tools.

DESIGN-ENTSCHEIDUNG:
Server 07 = Katalog (nur lesen)
Server 08 = Executor (ausfuehren)
-> Trennung von Concerns

VERFUEGBARE MCP-BAUSTEINE:
- Tools: (keine - nur Resources!)
- Resources: tests/catalog, tests/plan/{slug}
- Plaene: google-news

VORBEREITUNG
------------
Terminal 1: cd 07/server && dotnet run
Terminal 2: cd 07/client && dotnet run

Server laeuft auf http://localhost:5000/sse
KEIN LLM NOETIG.

SCHRITT-FUER-SCHRITT (MENUE-BASIERT)
------------------------------------

1. KATALOG LESEN
   Taste: 1 (Plaene listen)

   -> Zeige: Resource tests/catalog
   -> Erklaere: Markdown-Liste aller verfuegbaren Testplaene

2. EINZELNEN PLAN LESEN
   Taste: 2 (Plan lesen)
   Eingabe: google-news

   -> Zeige: Resource tests/plan/google-news
   -> Erklaere: Plan beschreibt Schritte, fuehrt sie NICHT aus

3. PLAN-INHALT VERSTEHEN
   Der Plan "google-news" enthaelt:
   - resolve-target: Ziel-URL bestimmen
   - http-get: GET senden
   - status-check: 2xx/3xx pruefen
   - content-fetch: Body lesen
   - content-check: Schlagwort suchen

   -> Erklaere: Das ist eine BESCHREIBUNG, keine Ausfuehrung

KERNAUSSAGE
-----------
"Dieser Server liefert nur Daten - keine Aktionen.
 Die Ausfuehrung passiert auf einem anderen Server.
 Das ist Separation of Concerns mit MCP."
